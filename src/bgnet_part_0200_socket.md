# ソケットとは？

「ソケット」という言葉をよく聞くと思いますが、正体が気になっている人も多いでしょう。要するにこういうものです：標準的な Unix の [i[File descriptor]] ファイル記述子を使って、他のプログラムとやり取りするための仕組みです。

え？

Unix ハッカーが「Unix では _なんでも_ ファイルだ！」と言うのを聞いたことがあるかもしれません。あれは、Unix プログラムが I/O を行うとき、すべてファイル記述子への読み書きで行う、という話です。ファイル記述子とは、開いているファイルに関連付けられた整数にすぎません。ただし（ここがポイント）、その「ファイル」はネットワーク接続、FIFO、パイプ、端末、ディスク上の実ファイルなど、ほぼなんでもあり得ます。Unix では本当になんでもファイルなのです！ だからインターネット越しに他のプログラムと通信するときも、ファイル記述子経由で行うことになります。信じられなくてもそうなのです。

「ネットワーク通信のファイル記述子はどこで手に入るんだ、先生？」——今はそんなこと考えていないかもしれませんが、答えておきます：[i[`socket()` function]] `socket()` システムルーチンを呼び出します。そうすると [i[Socket
descriptor]] ソケット記述子が返り、専用の [i[`send()` function]] `send()` と [i[`recv()` function]]
`recv()`（[`man send`](#sendman)、[`man recv`](#recvman)）ソケット呼び出しで通信します。

「でも待って！」と今頃叫んでいるかもしれません。「ファイル記述子なら、なぜ普通の
[i[`read()` function]] `read()` と [i[`write()` function]] `write()`
呼び出しでソケット通信できないんだ？」 短い答えは「できます！」 長い答えは「できますが、[i[`send()` function]]
`send()` と [i[`recv()` function]] `recv()` の方がデータ送信をはるかに細かく制御できる」です。

次は？ こういうことです：ソケットにはいろいろな種類があります。
DARPA インターネットアドレス（インターネットソケット）、ローカルノード上のパス名（Unix ソケット）、CCITT X.25 アドレス（無視してよい X.25 ソケット）など、使っている Unix の種類によって他にもあるでしょう。この文書で扱うのは最初のインターネットソケットだけです。


## インターネットソケットの 2 種類

え、インターネットソケットには 2 種類？ はい。いや、嘘です。もっとありますが、怖がらせたくなかっただけです。ここでは 2 種類だけ話します。この文だけ例外で、[i[Raw sockets]] 「Raw ソケット」も非常に強力なので調べてみる価値がある、と付け加えます。

さあ、2 種類とは？ 1 つは [i[Stream sockets]]
「ストリームソケット」、もう 1 つは [i[Datagram sockets]] 「データグラムソケット」で、以降は [i[`SOCK_STREAM` macro]]
「`SOCK_STREAM`」と [i[`SOCK_DGRAM` macro]] 「`SOCK_DGRAM`」
と呼ぶことにします。データグラムソケットは「コネクションレスソケット」とも呼ばれます。（本当にやりたければ [i[`connect()` function]] `connect()` もできます。[`connect()`](#connect) を参照。）

ストリームソケットは、信頼性のある双方向の接続型通信ストリームです。「1, 2」の順でソケットに 2 項目を出力すれば、向こう側でも「1, 2」の順で届きます。エラーもありません。そう確信しているので、誰かが違うと言おうものなら、耳を塞いで _ララララ_ と唱えます。

ストリームソケットを使う例？ [i[telnet]]
`telnet` や `ssh` を聞いたことがあるでしょう。これらはストリームソケットを使います。打った文字は打った順に届く必要がありますよね？ Web ブラウザも [i[HTTP
protocol]] ハイパーテキスト転送プロトコル（HTTP）でストリームソケットを使ってページを取得します。実際、ポート 80 の Web サイトに telnet して「`GET / HTTP/1.0`」と入力し RETURN を 2 回押せば、HTML が返ってきます！

> `telnet` が入っていない、入れたくない、または `telnet` が接続先にうるさい場合、このガイドには [flx[`telnot`|telnot.c]] という `telnet` 風プログラムが付属しています。ガイドの用途にはこれで十分でしょう。（telnet は実際には [flrfc[仕様化されたネットワークプロトコル|854]] で、`telnot` はそのプロトコルをまったく実装していません。）

ストリームソケットはどうやってこんなに高品質なデータ転送を実現するのか？ 「Transmission Control Protocol」、通称 [i[TCP]] 「TCP」（詳細は [flrfc[RFC 793|793]]）というプロトコルを使います。TCP はデータが順序通りかつエラーなく届くようにします。「TCP/IP」の TCP として聞いたことがあるかもしれません。IP 部分の [i[IP]] 「IP」は「Internet Protocol」（[flrfc[RFC 791|791]]）です。IP は主にインターネットルーティングを担当し、一般にデータの完全性までは保証しません。

[i[Datagram sockets]<]

さて、データグラムソケットは？ なぜコネクションレスと呼ばれるのか？ 要するにどういうこと？ なぜ信頼性がないのか？ 要点はこうです：データグラムを送れば、届く _かもしれない_。順不同で届く _かもしれない_。届いたなら、パケット内のデータ自体はエラーなしです。

データグラムソケットもルーティングに IP を使いますが、TCP は使わず「User Datagram Protocol」、通称 [i[UDP]] 「UDP」（[flrfc[RFC
768|768]]）を使います。

コネクションレスな理由？ 基本的に、ストリームソケットのように接続を維持する必要がないからです。パケットを組み立て、宛先情報付きの IP ヘッダを載せて送るだけ。接続は不要です。TCP スタックが使えない場合や、多少パケットが落ちても世界の終わりではない用途で使われます。例：`tftp`（FTP の小さな兄弟である trivial file transfer protocol）、`dhcpcd`（DHCP クライアント）、マルチプレイヤーゲーム、ストリーミング音声、ビデオ会議など。

[i[Datagram sockets]>]

「待てよ！ `tftp` や `dhcpcd` はホスト間でバイナリアプリケーションを転送するのに使われる！ 届いたときに動くならデータを失っちゃダメだろ！ 何の黒魔術だ？」

人間の友よ、`tftp` などは UDP の上に独自プロトコルを載せています。例えば tftp プロトコルでは、送られた各パケットに対し受信側が「受け取った！」（「ACK」パケット）を返す、と規定しています。送信側が例えば 5 秒以内に応答を得られなければ、ACK が来るまで再送します。この確認手順は、信頼性のある `SOCK_DGRAM` アプリケーションを実装するときに非常に重要です。

ゲーム、音声、動画など信頼性不要のアプリでは、落ちたパケットを無視するか、賢く補償します。（Quake プレイヤーなら、技術用語 _忌まわしきラグ_ でこの現象を知っているはずです。この場合の「忌まわしき」は、極めて下品な叫び声を表します。）

なぜ信頼性のない下位プロトコルを使うのか？ 理由は 2 つ：速度と速度。到着確認や順序保証を追うより、送りっぱなしの方がはるかに速いからです。チャットなら TCP が最適；プレイヤーの位置を秒間 40 回送るなら、1〜2 個落ちても大した問題ではなく、UDP が良い選択です。


## 低レベルの話とネットワーク理論 {#lowlevel}

プロトコルのレイヤリングに触れたので、ネットワークの実際の動きと、[i[`SOCK_DGRAM` macro]] `SOCK_DGRAM` パケットの組み立て例を説明します。実務上はこの節は飛ばしてもよいでしょう。背景知識としては有用です。

![データ・カプセル化。](dataencap.pdf "[Encapsulated Protocols Diagram]")

さあ子供たち、[i[Data encapsulation]] _データ・カプセル化_ の時間です！ これはとてもとても重要です。Chico State のネットワーク科目を履修すれば、たぶんこれを学ぶほど重要です `;-)`。要するにこういうことです：パケットが生まれ、最初のプロトコル（例：[i[TFTP]] TFTP）が [i[Data enacapsulation-->header]] ヘッダ（まれに [i[Data encapsulation-->footer]] フッタ）でパケットを包む（「カプセル化」）。次に全体（TFTP ヘッダ込み）を次のプロトコル（例：[i[UDP]] UDP）が再びカプセル化し、さらに [i[IP]]（IP）、最後にハードウェア（物理）層のプロトコル（例：[i[Ethernet]]
Ethernet）が包みます。

別のコンピュータがパケットを受信すると、ハードウェアが Ethernet ヘッダを剥がし、カーネルが IP と UDP ヘッダを剥がし、TFTP プログラムが TFTP ヘッダを剥がして、ようやくデータが取り出せます。

ここで悪名高い [i[Layered network model]]
[i[ISO/OSI]] _レイヤード・ネットワーク・モデル_（通称「ISO/OSI」）の話をします。このモデルは他のモデルに比べ多くの利点があるネットワーク機能の体系を述べています。例えば、データが物理的にどう伝わるか（シリアル、Thin Ethernet、AUI など）を気にせず、同じソケットプログラムを書けます。下位層のプログラムが面倒を見てくれるからです。実際のネットワークハードウェアやトポロジは、ソケットプログラマには透過的です。

早速、フル版モデルの各層を示します。ネットワークの試験用に覚えておいてください：

* Application（アプリケーション）
* Presentation（プレゼンテーション）
* Session（セッション）
* Transport（トランスポート）
* Network（ネットワーク）
* Data Link（データリンク）
* Physical（物理）

Physical Layer はハードウェア（シリアル、Ethernet など）です。Application Layer は物理層からできるだけ遠い——ユーザーがネットワークと対話する場所です。

このモデルは汎用的すぎて、本気なら自動車整備ガイドにも使えそうです。Unix により一貫したレイヤード・モデルはこうでしょう：

* Application Layer（_telnet、ftp など_）
* Host-to-Host Transport Layer（_TCP、UDP_）
* Internet Layer（_IP とルーティング_）
* Network Access Layer（_Ethernet、Wi-Fi など_）

ここまで来れば、これらの層が元データのカプセル化にどう対応するか見えてくるはずです。

単純なパケットを組み立てるのにこんなに手間がかかるのか！ しかも `cat` でパケットヘッダを自分で打ち込むのか！ 冗談です。
ストリームソケットなら [i[`send()` function]] `send()` でデータを送るだけ。データグラムソケットなら好きな方法でパケットをカプセル化して [i[`sendto()` function]]
`sendto()` で送るだけです。Transport Layer と Internet Layer はカーネルが、Network Access Layer はハードウェアが作ってくれます。現代技術の賜物です。

ネットワーク理論の短い寄り道はここまで。あ、ルーティングについて言いたかったことを全部言い忘れた：何も！ そう、まったく触れません。ルーターはパケットから IP ヘッダまで剥がし、ルーティングテーブルを参照し、[i[Blah blah blah]] _blah blah
blah_。本当に気になるなら [flrfc[IP RFC|791]] をどうぞ。学ばなくても、生きていけます。
